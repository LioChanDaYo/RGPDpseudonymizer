### Version Management

The version is defined in a single location: `pyproject.toml` line 3 (`version = "1.1.0"`). The PyInstaller spec file reads this dynamically (per Story 6.8 VERSION-001 fix). No `__version__` in `__init__.py` — the project uses `importlib.metadata` for runtime version queries.

**Critical:** Only update `pyproject.toml` version. Do NOT create a `__version__` variable unless the project already uses one (it doesn't). The `docs/v2.0-release-checklist.md` mentions updating `__init__.py` but this is **incorrect** — ignore that guidance.

[Source: `pyproject.toml:3`, `gdpr-pseudo-gui.spec` (VERSION-001 dynamic version read)]

### Release Workflow Architecture

Two GitHub Actions workflows coordinate on `v*` tag push:

1. **`release.yaml`** — Test → Publish to PyPI → Create GitHub Release (via `gh release create`)
2. **`build-executables.yaml`** — Build executables (4 platform jobs) → Upload to GitHub Release (via `gh release upload` with wait-for-release polling loop, max 10 min)

The `build-executables.yaml` `upload-release` job waits for the release created by `release.yaml` before uploading assets. This coordination is already implemented and tested (Story 6.8).

[Source: `.github/workflows/release.yaml`, `.github/workflows/build-executables.yaml`]
