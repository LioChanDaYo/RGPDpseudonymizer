# Story 7.4: Excel & CSV Format Support

## Status

**Done**

---

## Story

**As a** HR professional or compliance officer,
**I want** to pseudonymize Excel (.xlsx) and CSV files directly,
**so that** I can process employee data, case files, and research datasets without manual conversion to text.

---

## Acceptance Criteria

1. **AC1:** Excel file loading (.xlsx) via openpyxl
2. **AC2:** CSV file loading with auto-detection of delimiter and encoding
3. **AC3:** Cell-by-cell text extraction — each cell's text content is processed through the NER pipeline
4. **AC4:** Output options:
   - Pseudonymized Excel (.xlsx) — same structure, pseudonymized cell contents
   - Pseudonymized CSV — same structure, pseudonymized cell contents
   - Flattened text (.txt) — all cell contents concatenated (fallback)
5. **AC5:** Entity validation shows cell references (e.g., "Sheet1!B3") as context
6. **AC6:** GUI file dialog updated to include .xlsx, .csv filters
7. **AC7:** CLI `process` and `batch` commands accept .xlsx and .csv files
8. **AC8:** New dependencies as optional extras: `pip install gdpr-pseudonymizer[excel]`
9. **AC9:** Formulas are **not** preserved in pseudonymized output. Because openpyxl reads with `data_only=True` to access cached display values, formula strings are discarded. Pseudonymized `.xlsx` output contains static cell values only. This limitation must be documented in user-facing help and README.
10. **AC10:** Error handling: password-protected Excel, corrupt files, binary .xls (unsupported with clear message)
11. **AC11:** No regression in existing format support (.txt, .md, .pdf, .docx)
12. **AC12:** Unit and integration tests for Excel/CSV reading, processing, and output

### Cell-Aware Processing Design

The current pipeline assumes flat text with character offsets (`start_pos`/`end_pos`). For tabular files, each cell must be processed as an independent text unit:

1. **Read phase:** `read_excel_structured()` returns a `TabularDocument` with a list of `CellData` objects
2. **Detection phase:** Each cell's `value` is fed through the NER pipeline individually. Resulting `DetectedEntity` objects get `context_label = cell.cell_ref` and `start_pos`/`end_pos` relative to the cell text
3. **Validation phase:** The GUI/CLI validation shows `context_label` as context (e.g., "Sheet1!B3: Marie Dupont")
4. **Output phase:** Replacements are applied per-cell. The pseudonymized `TabularDocument` is written in the appropriate format (.xlsx, .csv, or .txt)

**Architect recommendation** (from Epic 7 Feasibility Review): "Add an optional `context_label: str | None` field to `DetectedEntity` rather than overloading `start_pos`/`end_pos`. This keeps the flat-text processing path unchanged while allowing cell-aware formats to attach location info. The cell-by-cell processing adapter should process each cell as an independent text unit, then tag resulting entities with their cell reference."
[Source: docs/prd/epic-7, Architect Feasibility Review]

### Security Considerations

- **No new attack surface beyond file I/O** — openpyxl parses XML-based .xlsx files. openpyxl 3.1+ handles XML security internally (no external `defusedxml` dependency required).
- **Encoding auto-detection** — CSV encoding detection tries UTF-8 first (safe), then falls back to Latin-1. No arbitrary code execution risk.
- **Formula injection** — openpyxl with `data_only=True` reads cached values, not formulas. Output writes plain values only. No formula injection risk in pseudonymized output.
- **File size** — Large Excel files with many sheets/rows could consume significant memory. Consider logging a warning for files with >100K cells.
- **Password-protected files** — openpyxl raises an exception for encrypted .xlsx; handle with clear user-facing error message.

### Technical Constraints

- **No new mandatory dependencies** — `openpyxl` is optional (`[excel]` extra). CSV uses Python stdlib only. [Source: epic-7 Implementation Notes]
- **`data_only=True`** — Required for openpyxl to read cached formula values instead of formula strings. Formula strings are discarded and not preserved in output (see AC9).
- **All public functions must have type hints.** [Source: architecture/19-coding-standards.md]
- **Absolute imports only.** [Source: architecture/19-coding-standards.md]
- **Quality gates**: black, ruff, mypy, pytest must all pass. [Source: architecture/19-coding-standards.md]
