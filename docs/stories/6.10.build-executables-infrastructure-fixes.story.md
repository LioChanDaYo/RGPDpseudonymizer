## Context

This is an **unplanned story** created to address infrastructure issues discovered during the v2.0.0 release (Story 6.9). The `build-executables.yaml` workflow was fixed for its `secrets`-in-`if` bug (Story 6.9), but all 4 individual build jobs fail due to pre-existing CI environment issues from Story 6.8. These are not application code bugs — they are CI runner configuration gaps.

**Previously tracked as:** "Story 6.8.1" in MEMORY.md, renumbered to 6.10 for clarity.

## Dev Notes

### Root Cause Analysis

All 4 failures are CI runner environment issues, not application code bugs:

| Platform | Failure | Root Cause | Fix |
|----------|---------|------------|-----|
| Windows | `makensis: command not found` | `windows-latest` does not include NSIS pre-installed | `choco install nsis -y` |
| macOS arm64 | `No space left on device` | spaCy model (~541MB) + PyInstaller output (~896MB) + DMG creation exceeds free disk | Remove unused pre-installed software (dotnet, android SDK, etc.) |
| macOS x86_64 | Runner `macos-13` unavailable | GitHub deprecated `macos-13` Intel runners | Migrate to `macos-14` (Apple Silicon) with Rosetta 2 cross-compilation (`arch -x86_64`) |
| Linux | `WARNING: Icon not found` | `create_appimage.sh` looks for icon at wrong path — PyInstaller 6.x puts data under `_internal/` | Update path to include `_internal/` prefix |

[Source: Story 6.9 Dev Agent Record lines 390-391, MEMORY.md "Open Items"]

### GitHub Actions Runner Context

- `windows-latest` → Windows Server 2022. NSIS is NOT included in the default image. Install via Chocolatey (`choco install nsis -y`).
- `macos-latest` / `macos-14` → macOS 14 (Sonoma) on Apple Silicon (M1). Provides ~14GB free disk. Rosetta 2 is pre-installed, enabling `arch -x86_64` cross-compilation.
- `macos-13` → **Deprecated.** GitHub Actions removed Intel-based macOS runners. `macos-14` and `macos-15` are Apple Silicon only.
- `ubuntu-22.04` → Ubuntu 22.04. Sufficient disk space. AppImage icon path is the only issue.

### Rosetta 2 Cross-Compilation (macOS x86_64)

A universal2 fat binary approach was investigated and ruled out — the spaCy/Explosion ecosystem (spaCy, thinc, cymem, preshed, murmurhash, blis) and NumPy only publish single-architecture wheels. PyInstaller's strict architecture validation aborts with "not a fat binary" errors.

Instead, the x86_64 build uses Rosetta 2 on Apple Silicon runners:
- `actions/setup-python@v5` installs universal2 Python for 3.11+ on macOS (contains both arm64 and x86_64 slices)
- Running under `arch -x86_64` forces the x86_64 Python slice; pip pulls x86_64 wheels for all native dependencies
- PyInstaller then produces a native x86_64 binary
- Build time is ~1.5-2x slower under emulation (acceptable for CI)

[Source: PO validation investigation — PyPI wheel analysis of spaCy, thinc, cymem, preshed, murmurhash, blis, numpy; PyInstaller docs on `--target-arch universal2`]

### PyInstaller 6.x `_internal/` Directory

PyInstaller 6.x changed the `--onedir` output structure. Data files and bundled packages are placed under `dist/<name>/_internal/` instead of directly in `dist/<name>/`. This means:

- **Correct path:** `dist/gdpr-pseudonymizer/_internal/gdpr_pseudonymizer/gui/resources/icons/icon_256.png`
- **Incorrect path:** `dist/gdpr-pseudonymizer/gdpr_pseudonymizer/gui/resources/icons/icon_256.png`

The NSIS installer already uses `_internal/` paths (fixed in Story 6.8 QA), but the Linux `create_appimage.sh` was missed.

[Source: Story 6.8 QA Finding NSIS-001, `gdpr-pseudo-gui.spec`]
