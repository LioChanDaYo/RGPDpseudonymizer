# Story 7.3: Neutral Pseudonym Theme

## Status

**Done**

---

## Story

**As a** user processing formal or legal documents,
**I want** a neutral/generic pseudonym theme that replaces names with identifiers like PER-001, LOC-001, ORG-001,
**so that** pseudonymized output is appropriate for contexts where themed names (Star Wars, LOTR) feel unprofessional.

---

## Acceptance Criteria

1. **AC1:** New theme option `--theme neutral_id` (CLI) and "Neutre / Générique" in GUI theme dropdown
2. **AC2:** Generates sequential identifiers per entity type: PER-001, PER-002, LOC-001, ORG-001, etc.
3. **AC3:** Compositional pseudonymization works — if "Marie Dupont" -> PER-001, then "Marie" -> PER-001-P (prenom) and "Dupont" -> PER-001-N (nom). If "Marie" also appears independently (not as a sub-entity of a compound name), it receives its own identifier (PER-002). Grouping follows existing variant grouping logic from Story 4.6.
4. **AC4:** Mapping table stores neutral pseudonyms like any other theme
5. **AC5:** Counter resets per database session (not globally)
6. **AC6:** Documentation updated (theme comparison table, CLI reference, GUI settings)
7. **AC7:** No regression in existing theme functionality (neutral, lotr, star_wars)
8. **AC8:** Unit tests for counter-based generation, compositional logic, and mapping persistence

### Task 1: Create `NeutralIdPseudonymGenerator` class (AC: 1, 2, 5)

- [x] **1.1:** Create a new class `NeutralIdPseudonymGenerator` in `gdpr_pseudonymizer/pseudonym/neutral_id_generator.py`. This class encapsulates counter-based pseudonym generation for the `neutral_id` theme.
  - State: `_counters: dict[str, int]` with keys `"PERSON"`, `"LOCATION"`, `"ORG"` — all starting at 0
  - Method `generate(entity_type: str) -> str`:
    - Increments the counter for `entity_type`
    - Returns formatted identifier: `PER-{counter:03d}`, `LOC-{counter:03d}`, `ORG-{counter:03d}`
    - Maps entity_type to prefix: `{"PERSON": "PER", "LOCATION": "LOC", "ORG": "ORG"}`
  - Method `reset() -> None` — resets all counters to 0
  - Method `set_counter(entity_type: str, value: int) -> None` — for restoring state from existing mappings
- [x] **1.2:** Add type hints to all public methods, use absolute imports, follow PascalCase class naming.

- [x] **2.4:** Implement `_assign_neutral_id_pseudonym()`. **IMPORTANT:** This method must NOT delegate to or fall through to the existing component-reuse lookup methods (`get_component_pseudonym()`, the `_component_mappings` scan loops at lines ~483-575) — those paths implement library-based reuse semantics that conflict with neutral_id behavior.     - Check `_component_mappings` for existing mappings of both components
    - If full name is new: generate next PER-NNN, store full mapping, derive component suffixes:
      - `_component_mappings[(first_name, "first_name")] = "PER-NNN-P"`
      - `_component_mappings[(last_name, "last_name")] = "PER-NNN-N"`

  - For PERSON with only one name (ambiguous): generate next PER-NNN
  - For LOCATION: generate next LOC-NNN
  - For ORG: generate next ORG-NNN
  - Return `PseudonymAssignment(pseudonym_full=identifier, pseudonym_first=..., pseudonym_last=..., theme="neutral_id", exhaustion_percentage=0.0)`
  - Exhaustion is always 0.0 — counter-based generation never exhausts
- [x] **2.5:** In `load_existing_mappings()`, when theme is `neutral_id`, scan existing pseudonyms to restore counters. Parse existing PER-NNN/LOC-NNN/ORG-NNN values, find the max counter for each type, and call `_neutral_id_generator.set_counter()` to resume from the correct point. ### Compositional Pseudonymization — neutral_id specifics

The existing compositional logic in `CompositionalPseudonymEngine` handles name parsing:
- "Marie Dupont" → `(first_name="Marie", last_name="Dupont")`
- Engine calls `manager.assign_pseudonym(entity_type="PERSON", first_name="Marie", last_name="Dupont")`

For `neutral_id`, the manager's `_assign_neutral_id_pseudonym()` method must:
1. Generate the full identifier: `PER-001`
2. Derive component identifiers: `PER-001-P` (prenom), `PER-001-N` (nom)
3. Store in `_component_mappings`: `{("Marie", "first_name"): "PER-001-P", ("Dupont", "last_name"): "PER-001-N"}`
4. If "Marie" appears independently later, it gets its OWN identifier `PER-002` (not `PER-001-P`)

This differs from library-based themes where standalone "Marie" reuses the same pseudonym component. For neutral_id, the PRD explicitly states: "If 'Marie' also appears independently, it receives its own identifier (PER-002)." This means component mappings are only used for recognizing sub-components of already-assigned compound names, NOT for reusing the component when the name appears alone.

### Security Considerations

- **No new attack surface** — neutral_id introduces no new inputs, file I/O, or network calls. It is a purely algorithmic transformation within the existing pseudonym pipeline.
- **Encryption at rest** — neutral_id pseudonyms (PER-001, etc.) are stored encrypted via the existing Fernet encryption layer, identical to other themes. No changes needed.
- **No sensitive data leakage** — counter-based identifiers contain no information about the original entity, which is inherently safe.
