# Story 6.8: Standalone Executables & Distribution

## Status

**Done**

---

## Story

**As a** non-technical user,
**I want** to download and run a single installer (.exe on Windows, .app on macOS),
**so that** I can use GDPR Pseudonymizer without installing Python, pip, or any dependencies.

---

## Acceptance Criteria

1. **AC1 — Packaging Tool Configured:**
   - PyInstaller 6.x selected and configured (per Story 6.1 / `gui-framework-decision.md` Section 7.1)
   - `--onedir` mode used (faster startup, LGPL compliance, easier debugging)
   - A `.spec` file created for reproducible builds

2. **AC2 — Windows Executable:**
   - `.exe` produced via PyInstaller `--onedir` mode
   - NSIS installer wrapping the directory into a single `.exe` installer (per Section 7.5)
   - Launches GUI by default; CLI available via command line (`gdpr-pseudo.exe` in install directory)
   - Tested on Windows 10 and Windows 11

3. **AC3 — macOS Application:**
   - `.app` bundle produced by PyInstaller
   - Wrapped in `.dmg` for distribution (using `hdiutil`)
   - Launches GUI by default
   - Tested on macOS 13+ (Intel and Apple Silicon)

4. **AC4 — Linux Package (Optional, Lower Priority):**
   - AppImage produced via `appimagetool`
   - Tested on Ubuntu 22.04+

5. **AC5 — Bundle Includes All Dependencies:**
   - Python runtime
   - All Python dependencies (SQLAlchemy, cryptography, structlog, etc.)
   - PySide6-Essentials Qt libraries
   - spaCy `fr_core_news_lg` model (~541MB)
   - Pseudonym libraries (`gdpr_pseudonymizer/resources/` — JSON files, YAML patterns)
   - GUI resources (icons, themes QSS files, i18n `.qm` files)

6. **AC6 — Bundle Size Target:**
   - Compressed installer < 500MB
   - Expected uncompressed: ~660-690MB (per `gui-framework-decision.md` Section 7.3)

7. **AC7 — Startup Time:**
   - Application starts in < 10 seconds on target hardware (first launch may be slower due to OS caching)

8. **AC8 — Code Signing (Preparation):**
   - Windows: Authenticode signing step in CI workflow (conditional on `WINDOWS_CERT` secret)
   - macOS: notarization step in CI workflow (conditional on `APPLE_DEVELOPER_ID` secret)
   - Unsigned builds produced when certificates are not available (beta testing)
   - Documentation notes the SmartScreen / Gatekeeper warnings for unsigned builds

9. **AC9 — CI/CD Pipeline:**
   - New GitHub Actions workflow `.github/workflows/build-executables.yaml`
   - Triggered on push to release tags (`v*`) — runs after existing `release.yaml` test job
   - Matrix: Windows (windows-latest), macOS (macos-latest), Linux (ubuntu-22.04)
   - Artifacts uploaded to GitHub Releases (attached to the same release created by `release.yaml`)

10. **AC10 — Installation Documentation:**
    - "Download and run" instructions for non-technical users added to README
    - Platform-specific installation notes (Windows SmartScreen, macOS Gatekeeper, Linux AppImage permissions)
    - Troubleshooting section (antivirus false positives, Gatekeeper warnings, missing system libs)

11. **AC11 — No Regression:**
    - PyPI installation path (`pip install gdpr-pseudonymizer`) continues to work unchanged
    - All existing tests pass (CLI + GUI)
    - Quality gates pass (Black, Ruff, mypy)
    - Coverage >= 86%

### Packaging Tool: PyInstaller 6.x

- **Selected tool:** PyInstaller 6.x (`--onedir` mode)
- **Why `--onedir` not `--onefile`:**
  - Faster startup (no temporary extraction step)
  - Easier debugging (files visible in directory)
  - LGPL compliance (Qt DLLs are separate files, satisfying relinking requirement)
  - Wrapped in platform installer (NSIS/DMG) provides single-file download experience

[Source: `architecture/gui-framework-decision.md#7.1-packaging-tool` and `#7.2-bundle-mode`]

### spaCy Model Bundling

The spaCy `fr_core_news_lg` model (~541MB) is the largest bundle component. It is located in the virtualenv under `site-packages/fr_core_news_lg/`. PyInstaller's `--add-data` flag should copy the entire model directory into the bundle.

**Frozen path resolution:** PyInstaller sets `sys._MEIPASS` in **both** `--onefile` and `--onedir` modes. In `--onefile` mode it points to a temporary extraction directory; in `--onedir` mode it points to the bundle's application directory. The `sys.frozen` attribute is also set in both modes. The application must resolve the spaCy model path at runtime using either:
1. Setting `spacy.util.get_data_path()` to the bundled model's parent directory, or
2. Using `spacy.load()` with an explicit path to the bundled model directory

**Important:** The model is loaded lazily (only when a document is first processed), so startup time is NOT affected by model size.

[Source: `architecture/3-tech-stack.md` (spaCy model spec), `architecture/gui-framework-decision.md#7.3`]

### Existing Release Workflow Integration

The existing `release.yaml` creates a GitHub Release via `gh release create`. The new `build-executables.yaml` should attach executables to the **same release** using `gh release upload`. Both workflows trigger on `v*` tags.

**Coordination:** Both workflows trigger on `v*` tags, so there is a race condition — the GitHub Release may not exist yet when `build-executables.yaml` tries to upload. The `upload-release` job must implement a **wait-for-release loop**:

```bash
# Wait up to 10 minutes for release.yaml to create the release
for i in $(seq 1 20); do
  if gh release view "$TAG" --json id >/dev/null 2>&1; then
    echo "Release found, uploading assets..."
    break
  fi
  echo "Release not found yet, waiting 30s... (attempt $i/20)"
  sleep 30
done
```

If the release is still not found after 10 minutes, the job should fail with a clear error message.

[Source: `.github/workflows/release.yaml` (verified)]

### Findings

#### CLI-001 — `console=False` breaks CLI on Windows (medium)

**File:** [gdpr-pseudo-gui.spec:189](gdpr-pseudo-gui.spec#L189)

The spec file sets `console=False`, making the executable a Windows GUI-subsystem application. On Windows, GUI-subsystem executables do **not** attach to the calling console — stdout/stderr output is silently discarded. This means `gdpr-pseudonymizer.exe --cli --help` (and all CLI operations) produce no visible output when run from cmd.exe or PowerShell.

**AC2 explicitly states:** "CLI available via command line (`gdpr-pseudo.exe` in install directory)" — this is not functional with `console=False`.

**Note:** The CI smoke test (`dist\gdpr-pseudonymizer\gdpr-pseudonymizer.exe --cli --help`) passes because it only checks exit code, not output visibility.

**Suggested fix:** Either (a) build a separate console-mode CLI executable alongside the GUI one, or (b) set `console=True` and accept the brief console flash on GUI launch, or (c) document the limitation and recommend users use `gdpr-pseudo` CLI from PyPI instead of standalone for CLI workflows.

#### CI-SEC-001 — Code signing steps never execute (medium)

**File:** [.github/workflows/build-executables.yaml:83](/.github/workflows/build-executables.yaml#L83), lines 164, 177, 187, 271, 284, 294

All code signing conditional steps use `if: env.WINDOWS_CERT_BASE64 != ''` (or similar) where the `env` variable is defined in the **same step's** `env:` block. In GitHub Actions, a step's `if` condition cannot reference the step's own `env` context — it only sees workflow-level and job-level env variables. Since no workflow/job-level env defines these, the condition always evaluates to `'' != ''` → `false`, and signing steps are **permanently skipped**, even when secrets are properly configured.

**Fix:** Replace `if: env.WINDOWS_CERT_BASE64 != ''` with `if: ${{ secrets.WINDOWS_CERT_BASE64 != '' }}` (and similarly for all macOS signing steps).

#### CI-SEC-002 — `notarytool --apple-id` receives wrong value (low)

**File:** [.github/workflows/build-executables.yaml:194](/.github/workflows/build-executables.yaml#L194)

The `xcrun notarytool submit --apple-id "$APPLE_DEVELOPER_ID"` passes the codesigning identity string (e.g., "Developer ID Application: Company (TEAMID)") as the `--apple-id` argument. But `notarytool --apple-id` expects an **Apple ID email address** (e.g., `dev@example.com`). This would cause notarization to fail. Latent bug — blocked by CI-SEC-001.

**Fix:** Add a separate `APPLE_ID_EMAIL` secret and use it for `--apple-id`.

#### NSIS-001 — Installer icon path likely incorrect for PyInstaller 6.x (medium)

**File:** [installer/windows/installer.nsi:43-44](installer/windows/installer.nsi#L43-L44)

The NSIS `MUI_ICON` and `MUI_UNICON` reference `${DISTDIR}\gdpr_pseudonymizer\gui\resources\icons\app.ico`. However, PyInstaller 6.x `--onedir` mode places data files under `_internal/`, so the actual icon location would be `${DISTDIR}\_internal\gdpr_pseudonymizer\gui\resources\icons\app.ico`. The integration test at [test_pyinstaller_build.py:93-103](tests/integration/test_pyinstaller_build.py#L93-L103) confirms icons are under `_internal/`. If this path is wrong, `makensis` will fail with a file-not-found error.

**Fix:** Update paths to `${DISTDIR}\_internal\gdpr_pseudonymizer\gui\resources\icons\app.ico`.

#### TEST-001 — Two main() dispatch tests are no-ops (medium)

**File:** [tests/unit/test_standalone_entry.py:200-239](tests/unit/test_standalone_entry.py#L200-L239)

- `test_gui_mode_default`: Sets up mock contexts but the innermost block is `pass` — `main()` is never called. The test passes without testing anything.
- `test_cli_mode_removes_cli_flag`: Performs manual list manipulation to test `--cli` removal instead of calling `main()`. Never exercises the actual code path.

Both tests appear in the test count (31 tests) but provide zero coverage of the `main()` function's dispatch behavior.

**Fix:** Rewrite both tests to actually call `standalone_entry.main()` and assert on the mocked CLI/GUI function invocations.

#### VERSION-001 — Hardcoded version in macOS bundle (low)

**File:** [gdpr-pseudo-gui.spec:228](gdpr-pseudo-gui.spec#L228)

`CFBundleShortVersionString: "1.1.0"` is hardcoded. Future releases will show "1.1.0" in macOS system info regardless of actual version. Should read version from package metadata or accept it as a build parameter.
